# DPD v0.3.0 → v0.3.1 Migration Guide

**Audience**: Developer migrating an existing v0.3.0 DB or integrating updated skill files.
**Date**: 2026-05-22

---

## 1. DB Schema Migration

Migration is **automatic** — `Storage.open()` detects `PRAGMA user_version = 3` and applies
`migrate_v3_to_v4.py` before returning the connection. No manual step is required for normal use.

To run the migration script directly on an existing DB:

```bash
python -m dpd_mcp_server.migrate_v3_to_v4 path/to/your.db
```

### Schema changes summary

| Table | Change | Notes |
|-------|--------|-------|
| `sessions` | `mode TEXT` column added (nullable) | Existing rows → `NULL` (legacy heuristic) |
| `pool_items` | `rejected_at TEXT` column added (nullable) | `NULL` = not rejected |
| `pool_items` | `rejected_reason TEXT` column added (nullable) | User's rejection text |
| `pool_items` | `text_hash TEXT` column added (nullable) | Dedup key |
| `nodes` | `provenance TEXT NOT NULL DEFAULT 'grounded'` added | CHECK constraint (see §6) |
| `nodes` | `CHECK(provenance IN ('grounded','inferred','imported','manual'))` | Raises IntegrityError on invalid value |
| `PRAGMA user_version` | `3` → `4` | Migration guard |

### Reverse migration

**Not supported.** v0.3.1 → v0.3.0 downgrade is a one-way operation. Back up your DB before upgrading if rollback might be needed.

## 2. User-Visible Behavior Change

| Aspect | v0.3.0 (active mode) | v0.3.1 (ambient overlay) |
|--------|---------------------|--------------------------|
| Invocation model | User deliberately calls each graph tool | Claude observes conversation and proposes updates at natural pauses |
| `/dpd` trigger | Opens interactive session; user drives every operation | Entry phase: Claude elicits goal → graph bootstrap; then hands off to ambient |
| Cognitive overhead | Per-operation confirmation | Goal confirmation (entry) + lightweight proposal confirmation (ambient) |
| Session lifecycle | Single active phase | `entry` → `ambient` → `idle` (resumable) |

### Existing session resume

Sessions created under v0.3.0 have `mode = NULL` after migration. On `/dpd` resume, Claude
detects `mode = NULL` and applies heuristic (spec §6.1):

- If Pool is non-empty or active subgraph exists → suggest ambient resume
- Otherwise → suggest entry phase for a new subgraph

Claude presents a suggestion; the user confirms before ambient mode begins. Silent auto-resume
is intentionally absent (per spec §3.5).

---

## 3. Backward Compatibility

- **Legacy sessions** (`mode = NULL`) continue to work — heuristic detection per spec §6.1.
- **All v0.3.0 MCP tool signatures are unchanged.** New arguments are optional with safe defaults.
- **Old skill behavior** maps to v0.3.1 entry phase + manual graph operations — no functional
  regression for users who prefer explicit tool invocation.

---

## 4. New MCP Tools in v0.3.1

| Tool | Purpose |
|------|---------|
| `pool_reject(pool_id, reason?)` | Soft-suppress a pool item (sets `rejected_at` + `rejected_reason`). Orthogonal to `pool_drop` (physical removal). |
| `set_session_mode(session_id, mode)` | Explicit `entry` / `ambient` / `idle` transition per spec §9.1.1. |
| `bulk_import_subgraph(session_id, root_id, nodes, edges, provenance, state)` | Atomic multi-node + edge import for `/dpd-import`. Inserts in a single transaction. |

Extended tool arguments (existing tools, new optional params):

| Tool | New arg | Default | Purpose |
|------|---------|---------|---------|
| `pool_list` | `include_rejected` | `false` | Include rejected items in results |
| `pool_list` | `rejected_only` | `false` | Return only rejected items |
| `add_node` | `provenance` | `'grounded'` | Source category for the new node |
| `add_node` | `state` | `'active'` | Initial node state (`/dpd-import` passes `'archived'`) |
| `start_session` | `mode` | `'entry'` | Initial session mode |

## 5. New Sub-Skills in v0.3.1

Six sub-skills ship alongside the rewritten `/dpd` main skill:

| Skill | Purpose |
|-------|---------|
| `/dpd-status` | Show current graph + Pool + in-session pending updates |
| `/dpd-dump` | Export full graph as YAML or Mermaid via `export_yaml` / `export_mermaid` |
| `/dpd-edit <node\|pool_id>` | Manual node or pool item mutation (workaround for missing `pool_unreject` — see §7) |
| `/dpd-summary-md` | Render closed/decided nodes as a clean markdown summary |
| `/dpd-fill` | Propose inferred nodes for detected gaps; user opts in per node |
| `/dpd-import` | Import external prose/spec/graph as a hypothetical archived DPD subgraph; pairs with `/dpd-fill` + `/fcot` for gap analysis |

## 6. Schema CHECK Constraints

`nodes.provenance` is now enforced at the DB level:

```sql
CHECK(provenance IN ('grounded', 'inferred', 'imported', 'manual'))
```

Inserting or updating a node with any other provenance value raises a `sqlite3.IntegrityError`.
Existing v0.3.0 nodes are migrated to `'grounded'`.

## 7. Known Limitations / Deferred to v0.3.2+

From spec §10:

| Item | Reason deferred |
|------|----------------|
| `/dpd-trace` — audit history surface | Conversational observation sufficient for initial dogfood |
| Advanced `/dpd-fill` goal-driven auto decomposition | Base `/dpd-fill` usage pattern must stabilize first |
| `/fcot` as internal pre-verification for every proposal | Conflicts with ambient overlay philosophy (over-verification) |
| Tangled-conversation auto-detection threshold tuning | Base heuristics (§2.2) need dogfood data |
| Pool item lifecycle automation (auto-drop after N days) | Rejected lifecycle semantics not yet stable |
| Cross-session goal linking | Single-session loop must stabilize first |

Additional gaps discovered during dogfood:

- **No `pool_unreject` tool**: Workaround — `pool_drop` + `pool_add` to re-add the item.
  Documented in `/dpd-edit` SKILL.md. `pool_unreject` is tracked for v0.3.2.
- **No `update_node_text` tool**: Workaround — spawn a replacement node with the corrected text
  and link via `supersedes` edge. Tracked for v0.3.2.
